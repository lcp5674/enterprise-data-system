## 1 总则

### 1.1 目的

本标准旨在建立统一的数据资产分类分级体系，为数据安全保护提供依据，确保数据资产得到适当的保护和管理。

### 1.2 适用范围

本标准适用于企业所有数据资产，包括：
- 结构化数据（数据库表、视图）
- 半结构化数据（JSON、XML、日志）
- 非结构化数据（文档、图片、音视频）
- API接口数据
- 流式数据

### 1.3 法规依据

| 法规/标准 | 适用范围 |
|-----------|----------|
| 《数据安全法》 | 数据安全保护基本要求 |
| 《个人信息保护法》 | 个人信息处理规范 |
| 《网络安全法》 | 网络安全等级保护 |
| GB/T 35273 | 个人信息安全规范 |
| ISO 27001 | 信息安全管理体系 |
| GDPR | 跨境数据传输（可选） |

## 2 数据分类标准

### 2.1 分类维度

数据分类采用多维度划分：

| 分类维度 | 说明 |
|----------|------|
| 业务域 | 按业务领域分类 |
| 数据主体 | 按数据涉及的主体分类 |
| 用途 | 按数据使用目的分类 |
| 来源 | 按数据产生来源分类 |

### 2.2 业务域分类

| 编码 | 业务域 | 说明 | 示例 |
|------|--------|------|------|
| CRM | 客户关系管理 | 客户相关数据 | 客户信息、商机、合同 |
| ERP | 企业资源计划 | 内部运营数据 | 财务、采购、库存 |
| SCM | 供应链管理 | 供应链相关数据 | 供应商、物流、库存 |
| HRM | 人力资源管理 | 员工相关数据 | 员工档案、薪酬、考勤 |
| PRD | 生产研发 | 产品研发数据 | 设计文档、代码、技术专利 |
| MKT | 市场营销 | 营销相关数据 | 推广活动、广告投放、转化数据 |
| FIN | 财务管理 | 财务相关数据 | 凭证、报表、预算 |
| OPR | 运维管理 | IT运维数据 | 系统日志、监控数据 |
| GOV | 公司治理 | 合规治理数据 | 审计记录、合规报告 |
| PUB | 公共数据 | 可对外公开数据 | 产品介绍、新闻公告 |

### 2.3 数据主体分类

| 分类 | 定义 | 敏感程度 |
|------|------|----------|
| 个人数据 | 涉及自然人个人信息 | 高 |
| 企业数据 | 涉及企业商业信息 | 中-高 |
| 公共数据 | 可公开获取的信息 | 低 |
| 混合数据 | 个人+企业混合数据 | 高 |

## 3 数据分级标准

### 3.1 分级体系

采用四级分级体系，与《数据安全法》三级保护要求对应：

| 等级 | 名称 | 保护要求 | 典型数据 |
|------|------|----------|----------|
| L1 | 公开级 | 公开披露 | 产品资料、新闻公告 |
| L2 | 内部级 | 内部使用 | 内部流程、组织架构 |
| L3 | 敏感级 | 严格保护 | 客户隐私、商业机密 |
| L4 | 机密级 | 极度严格 | 核心机密、密钥 |

### 3.2 分级定义

#### L1 公开级（公开数据）

**定义**：可向社会公众公开，无任何使用限制的数据。

**保护要求**：
- 无需加密存储
- 无需访问控制
- 可自由传播

**识别标准**（满足任一）：
- 已公开发布
- 可从公开渠道获取
- 不涉及任何敏感信息

**示例数据**：
- 公司官网产品介绍
- 公开的招聘信息
- 已发布的新闻稿
- 产品说明书

---

#### L2 内部级（内部数据）

**定义**：仅限企业内部人员使用，泄露可能对企业造成一般影响。

**保护要求**：
- 存储：一般加密（可选）
- 访问：需登录认证
- 传输：建议加密
- 操作：记录审计日志

**识别标准**（满足任一）：
- 涉及企业内部信息
- 泄露可能影响企业形象
- 竞争对手可能感兴趣
- 内部流程文档

**示例数据**：
- 组织架构信息
- 内部工作流程
- 员工通讯录（非敏感）
- 内部培训资料
- 项目计划（不含敏感内容）

---

#### L3 敏感级（敏感数据）

**定义**：涉及个人隐私或商业秘密，泄露可能对个人权益或企业利益造成严重影响。

**保护要求**：
- 存储：必须加密
- 访问：需申请+审批
- 传输：必须加密
- 操作：详细审计日志
- 脱敏：导出时必须脱敏

**识别标准**（满足任一）：
- 涉及个人身份信息（姓名+联系方式+地址等）
- 涉及个人财务信息（银行账号、信用卡等）
- 涉及个人健康信息
- 涉及未公开的商业信息
- 涉及用户行为数据

**示例数据**：
- 客户姓名+手机号+地址
- 用户身份证号
- 员工薪资信息
- 客户订单信息
- 财务报表（未公开）
- 客户名单、联系方式
- 用户行为日志

---

#### L4 机密级（核心机密）

**定义**：企业核心机密，泄露可能对企业生存造成致命影响。

**保护要求**：
- 存储：强制加密+隔离存储
- 访问：最高级别审批+多因素认证
- 传输：加密+专线
- 操作：实时监控+异常告警
- 备份：加密备份+异地存储

**识别标准**（满足任一）：
- 涉及企业核心商业机密
- 涉及核心技术/算法
- 涉及密钥/证书
- 涉及战略规划
- 涉及重大交易信息

**示例数据**：
- 系统登录密码
- 数据库连接密钥
- 加密算法/密钥
- 战略规划文档
- 并购重组信息
- 核心技术源代码
- 核心算法模型

## 4 分级判定规则

### 4.1 自动分级规则

系统根据规则自动进行分级预判：

```yaml
# 自动分级规则示例
rules:
  - name: "身份证号检测"
    pattern: "(身份证|ID_CARD)"
    condition: "字段名匹配"
    result: "L3"
    
  - name: "手机号检测"
    pattern: "(手机|PHONE|MOBILE)"
    condition: "字段名匹配"
    result: "L3"
    
  - name: "密码检测"
    pattern: "(密码|PASSWORD|SECRET)"
    condition: "字段名匹配"
    result: "L4"
    
  - name: "密钥检测"
    pattern: "(密钥|KEY|SECRET|TOKEN)"
    condition: "字段名匹配"
    result: "L4"
```

### 4.2 组合字段判定

多个低敏感字段组合可能形成高敏感数据：

| 组合条件 | 判定结果 |
|----------|----------|
| 姓名 + 手机号 | L3 |
| 姓名 + 身份证号 | L3 |
| 姓名 + 地址 | L3 |
| 姓名 + 手机号 + 地址 | L3 |
| 姓名 + 银行卡号 | L3 |

### 4.3 分级调整

| 调整类型 | 说明 | 审批级别 |
|----------|------|----------|
| 升级 | 低级调整为高级 | 业务负责人 |
| 降级 | 高级调整为低级 | 数据安全负责人 |
| 紧急调整 | 业务紧急需要 | CIO/CISO审批 |

## 6 分级保护矩阵

### 6.1 访问控制矩阵

| 操作 | L1公开 | L2内部 | L3敏感 | L4机密 |
|------|--------|--------|--------|--------|
| 查看 | 所有用户 | 认证用户 | 授权用户 | 申请+审批 |
| 编辑 | 授权用户 | 授权用户 | 负责人 | 多重审批 |
| 删除 | 管理员 | 管理员 | 负责人+审批 | 不允许 |
| 导出 | 所有人 | 认证用户 | 脱敏+审批 | 禁止 |
| 下载 | 所有人 | 认证用户 | 禁止 | 禁止 |

### 6.2 加密要求

| 级别 | 存储加密 | 传输加密 | 备份加密 |
|------|----------|----------|----------|
| L1 | 不需要 | 可选 | 不需要 |
| L2 | 可选(AES-128) | TLS 1.2 | 可选 |
| L3 | 必须(AES-256) | TLS 1.3 | 必须 |
| L4 | 必须(AES-256+HSM) | TLS 1.3+专线 | 必须+异地 |

### 6.3 脱敏规则

| 级别 | 脱敏类型 | 规则 |
|------|----------|------|
| L1 | 不脱敏 | - |
| L2 | 可选脱敏 | 按需 |
| L3 | 必须脱敏 | 部分遮蔽 |
| L4 | 完全脱敏 | 替换/哈希 |

**脱敏规则示例**：

| 字段 | L3脱敏规则 | 效果 |
|------|-----------|------|
| 手机号 | 保留前3后4 | 138****5678 |
| 身份证 | 保留前6后4 | 110101****0011 |
| 姓名 | 保留姓氏 | 张** |
| 银行卡 | 保留前6后4 | 622202****1234 |
| 邮箱 | @前保留1 | a***@xxx.com |
| 地址 | 保留省市区 | 北京市朝阳区*** |

### 7.2 定级审批规则

| 初始级别 | 需审批 | 审批人 | 时效 |
|----------|--------|--------|------|
| L1 | 否 | - | 即时 |
| L2 | 否 | - | 即时 |

| L4 | 是 | 数据Owner+CISO | 5个工作日 |

### 7.3 定级变更

| 变更类型 | 重新审批 | 说明 |
|----------|----------|------|
| 字段内容变化 | 是 | 内容变化可能导致级别变化 |
| 数据范围扩大 | 是 | 涉及更多敏感数据 |
| 使用场景变化 | 是 | 可能影响分级 |
| 例行复核 | 是 | 每年至少复核一次 |

## 8 管理职责

### 8.1 角色定义

| 角色 | 职责 |
|------|------|
| 数据Owner | 对数据资产安全负责，审批数据分级 |
| 数据Keeper | 执行日常数据管理，包括分级维护 |
| 数据User | 使用数据，遵守分级保护要求 |
| 安全专员 | 监督数据分级执行，处理分级争议 |
| CISO | 最终审批L4级别数据分级 |

### 8.2 安全责任矩阵（RACI）

| 活动 | 数据Owner | 数据Keeper | 数据User | 安全专员 | CISO |
|------|-----------|------------|----------|----------|------|
| 初始分级 | R | A | I | C | I |
| 分级审批 | A | R | I | C | R(L4) |
| 分级复核 | A | R | I | C | I |
| 分级调整 | A | R | - | C | R(L4) |
| 监督执行 | I | I | I | R | A |

**R=执行 A=审批 C=咨询 I=知情**

## 3. OWASP Top 10测试用例

### 3.1 A01 访问控制失效

```markdown
## A01:2021 - 访问控制失效测试

### 测试用例1：水平越权
```
测试步骤：
1. 使用用户A登录系统
2. 获取用户A可访问的资产列表
3. 尝试修改URL中的资产ID为用户B的资产ID
4. 尝试访问/修改用户B的资产详情

预期结果：
- 用户A无法访问用户B的私有资产
- 返回403 Forbidden或404 Not Found

测试脚本：
GET /api/v1/assets/12345  # 12345是用户B的资产
Headers: Authorization: Bearer {UserA_Token}

预期响应：
HTTP/1.1 403 Forbidden
{
  "code": 403,
  "message": "无权访问此资源"
}
```

### 测试用例2：垂直越权
```
测试步骤：
1. 使用普通用户账号登录
2. 尝试访问管理员API
   GET /api/v1/admin/users
   GET /api/v1/admin/config
   POST /api/v1/admin/system/reset
3. 尝试执行管理操作

预期结果：
- 普通用户无法访问管理员接口
- 返回401 Unauthorized或403 Forbidden
```

### 测试用例3：IDOR漏洞
```
测试步骤：
1. 获取资产ID列表
2. 遍历尝试访问不同ID
3. 检查是否有未授权访问

测试参数：
- /api/v1/assets/{id}
- /api/v1/assets/{id}/download
- /api/v1/assets/{id}/export
```

### 测试用例4：直接对象引用
```
测试步骤：
1. 获取当前用户资源列表
2. 获取资源关联的文件ID
3. 尝试通过文件ID直接下载
   GET /api/v1/files/file-12345

预期结果：
- 只有资源所有者才能下载关联文件
- 返回403 Forbidden
```

### 测试用例5：API访问控制
```
测试步骤：
1. 枚举API端点
2. 测试未认证访问
   GET /api/v1/assets
3. 测试认证后访问
   GET /api/v1/assets
   Headers: Authorization: Bearer {token}
4. 测试权限升级
   GET /api/v1/admin/**
```

### 3.3 A03 注入

```markdown
## A03:2021 - 注入测试

### 测试用例1：SQL注入
```
测试步骤：
1. 识别所有输入点
2. 尝试SQL注入Payload

基本测试：
- ' OR '1'='1
- ' OR 1=1--
- " OR "1"="1
- ' UNION SELECT NULL--
- ' AND SLEEP(5)--

盲注测试：
- ' AND 1=1--
- ' AND 1=2--
- ' AND SUBSTRING(@@version,1,1)='5'

时间盲注：
- ' AND IF(1=1,SLEEP(5),0)--
- '; WAITFOR DELAY '00:00:05'--

测试端点：
- GET /api/v1/assets?name=' OR '1'='1
- POST /api/v1/assets/search
  {"name": "' OR '1'='1"}
- GET /api/v1/assets?type=TABLE'--

预期结果：
- 返回错误或空结果
- 不应返回所有数据
- 不应有SQL错误信息泄露

## 5. 漏洞修复与验证

### 5.1 漏洞分级标准

```markdown
## 漏洞分级标准

### 严重（Critical）
- 直接获取服务器权限
- 大量敏感数据泄露
- 认证机制完全绕过

### 高危（High）
- 敏感数据泄露
- 权限绕过
- SQL注入
- 远程代码执行

### 中危（Medium）
- XSS存储型
- CSRF
- 暴力破解无限制
- 信息泄露

### 低危（Low）
- XSS反射型
- 弱密码策略
- 提示信息泄露
```

### 5.2 漏洞修复报告模板

```markdown
# 漏洞修复报告

## 漏洞信息
- **漏洞编号**：VULN-2024-001
- **漏洞名称**：SQL注入漏洞
- **风险等级**：高危
- **发现日期**：2026-04-10
- **修复日期**：2026-04-15

## 漏洞详情
### 位置
/api/v1/assets/search?name={input}

### 漏洞描述
用户输入的name参数未经过滤直接拼接到SQL查询中，
攻击者可利用此漏洞获取数据库敏感信息。

### 漏洞验证
```bash
curl -X GET "/api/v1/assets/search?name=' OR '1'='1"
# 返回所有资产数据
```

## 修复方案
### 修复代码
```java
// 修复前
@Query("SELECT a FROM DataAsset a WHERE a.assetName LIKE '%" + name + "%'")

// 修复后
@Query("SELECT a FROM DataAsset a WHERE a.assetName LIKE %:name%")
List<DataAsset> searchByName(@Param("name") String name);
```

### 修复说明
1. 使用参数化查询替代字符串拼接
2. 添加输入参数校验
3. 部署WAF规则

## 修复验证
### 验证日期：2026-04-16
### 验证人员：安全团队
### 验证结果：✅ 通过

### 验证步骤
1. 重新执行注入测试 - ❌ 注入失败
2. 检查代码规范 - ✅ 使用参数化查询
3. 回归测试 - ✅ 功能正常

## 遗留问题
无
```

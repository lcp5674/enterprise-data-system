### 2.2 前后端API路径严重不匹配

#### 问题1：SSO Controller缺失

**前端**（constants/index.ts）：
```typescript
SSO: {
  LOGIN: '/auth/api/v1/sso/login',        // ❌ 错误路径
  CALLBACK: '/auth/api/v1/sso/callback',
  USERINFO: '/auth/api/v1/sso/userinfo',
  // ...
}
```

**后端**（edams-auth）：**没有SSOController！**

- AuthController 的路径是 `/api/v1/auth`
- 但前端 SSO 路径是 `/auth/api/v1/sso/...`（多了 `/auth` 前缀）
- Gateway 路由是 `/api/v1/auth/**`，无法匹配 `/auth/api/v1/sso/...`

**影响**：所有SSO登录功能（企业微信、钉钉、LDAP）全部失效！

#### 问题2：治理引擎API路径缺少版本号

**前端**：
```typescript
GOVERNANCE: {
  TASKS: {
    EXECUTE: (id) => `/api/v1/governance/tasks/${id}/execute`,  // ✓ 有v1
    SUB_TASKS: (id) => `/api/v1/governance/tasks/${id}/sub-tasks`,
  }
}
```

**后端**（GovernanceTaskController）：
```java
@RequestMapping("/api/governance/tasks")  // ❌ 缺少 /v1
```

**影响**：治理任务执行接口404。

#### 问题3：密码修改路径不匹配

**前端**：
```typescript
AUTH: {
  CHANGE_PASSWORD: '/api/v1/auth/password/change',  // POST
}
```

**后端**（AuthController）：
```java
@PutMapping("/password")  // PUT，路径也不一样
```

**影响**：修改密码功能失效。

#### 问题4：API路径汇总对比

| 功能模块 | 前端路径 | 后端路径 | 状态 |
|---------|---------|---------|------|
| SSO登录 | `/auth/api/v1/sso/login` | **缺失Controller** | 🔴 404 |
| SSO回调 | `/auth/api/v1/sso/callback` | **缺失Controller** | 🔴 404 |
| 修改密码 | `/api/v1/auth/password/change` | `/api/v1/auth/password` (PUT) | 🔴 404 |
| 治理任务执行 | `/api/v1/governance/tasks/{id}/execute` | `/api/governance/tasks/{id}/execute` | 🔴 404 |
| 治理子任务 | `/api/v1/governance/tasks/{id}/sub-tasks` | `/api/governance/tasks/{id}/sub-tasks` | 🔴 404 |
| 登录 | `/api/v1/auth/login` | `/api/v1/auth/login` | ✅ 匹配 |
| 资产列表 | `/api/v1/assets` | `/api/v1/assets` | ✅ 匹配 |
| 用户列表 | `/api/v1/users` | `/api/v1/users` | ✅ 匹配 |

## 三、P1重要问题

### 3.1 后端Controller包名不一致

| 服务 | 包名 | 问题 |
|------|-----|------|
| edams-parent/* | `com.enterprise.edams.*` | 标准 |
| governance-engine | `com.enterprise.dataplatform.governance` | ❌ 不一致 |
| quality-service | `com.enterprise.dataplatform.*` | ❌ 不一致 |
| standard-service | `com.enterprise.dataplatform.*` | ❌ 不一致 |

**影响**：统一封装、日志追踪、监控标识不一致。

### 3.2 后端响应格式不一致

| 服务 | 响应格式 | 问题 |
|------|---------|------|
| edams-parent/* | `Result<T>` | 标准 |
| governance-engine | `ResponseEntity<T>` | ❌ 不一致 |
| quality-service | `ResponseEntity<T>` | ❌ 不一致 |

**影响**：前端需要处理两种响应格式，增加复杂度。

## 四、代码质量评估

### 4.1 Controller实现对比

**良好的实现**（edams-auth/AuthController）：
- ✅ 使用 `@Tag` 注解提供 Swagger 文档
- ✅ 使用 `@Operation` 注解描述每个接口
- ✅ 使用 `@Valid` 进行参数校验

- ❌ 使用 `ResponseEntity<T>` 而非统一格式
- ❌ 部分方法缺少 `@Operation` 描述

```yaml

  port: 8106  # 改为 8106
```

#### 修复2：创建SSOController

在 `edams-auth` 中创建 `SSOController.java`：

```java
@RestController
@RequestMapping("/api/v1/sso")
public class SSOController {
    // 实现所有SSO相关接口
}
```

#### 修复3：修复治理引擎API路径

修改 `GovernanceTaskController.java`：

```java
@RequestMapping("/api/v1/governance/tasks")  // 添加 /v1
```

#### 修复4：修复密码修改路径

修改 `AuthController.java` 或 `constants/index.ts`，统一为：
- 路径：`/api/v1/auth/password`
- 方法：`POST`

2. 统一响应格式为 `Result<T>`

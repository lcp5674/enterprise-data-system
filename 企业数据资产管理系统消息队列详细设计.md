## 1 消息队列架构概述

### 1.1 消息队列在系统中的作用

消息队列是企业数据资产管理系统实现异步通信、事件驱动、服务解耦的核心基础设施。通过消息队列，系统实现以下核心能力：

- **异步处理**：非关键路径业务异步执行，提升系统响应速度
- **服务解耦**：生产者和消费者解耦，支持独立演化和扩展
- **流量削峰**：缓冲突发流量，保护下游服务
- **事件驱动**：支持复杂的业务流程编排和事件溯源
- **最终一致性**：通过消息传递实现分布式系统最终一致性

### 1.2 消息队列选型

| 特性 | Apache Kafka | RabbitMQ | Apache Pulsar |
|------|--------------|----------|---------------|
| 吞吐量 | 百万级/秒 | 万级/秒 | 百万级/秒 |
| 延迟 | 毫秒级 | 微秒级 | 毫秒级 |
| 消息持久化 | 支持 | 支持 | 支持 |
| 消息回溯 | 支持 | 不支持 | 支持 |
| 多租户 | 支持 | 有限支持 | 原生支持 |
| 消息优先级 | 不支持 | 支持 | 不支持 |
| 事务消息 | 支持 | 支持 | 支持 |
| 适用场景 | 大数据、日志、流处理 | 中小型系统、灵活路由 | 云原生、多租户 |

本系统选择 **Apache Kafka** 作为核心消息队列，因其高吞吐、消息回溯、多消费者支持等特性更适合数据资产管理系统的复杂场景。

## 4 消息格式设计

所有Topic的消息采用统一的Envelope格式：

```json
{
  "envelope": {
    "message_id": "msg_uuid_123456",
    "correlation_id": "corr_789012",
    "causation_id": "cmd_345678",
    "timestamp": "2026-04-10T12:30:00.000Z",
    "message_type": "DomainEvent",
    "schema_version": "1.0",
    "content_type": "application/json"
  },
  "headers": {
    "tenant_id": "tenant_001",
    "user_id": "user_001",
    "source_service": "metadata-service",
    "trace_id": "trace_abc123",
    "span_id": "span_def456",
    "priority": "normal",  // low | normal | high | critical
    "ttl": 86400  // 秒，消息过期时间
  },
  "body": {
    // 具体业务消息内容
  }
}
```

### 4.2 Schema Registry配置

```yaml
# Schema Registry配置
schema:
  # 兼容性模式
  compatibility: BACKWARD
  
  # 演进的版本管理
  versioned: true
  
  # 自动注册
  auto_register: false
  
# 支持的序列化格式
serialization:
  json:
    enabled: true
    strict_validation: true
  avro:
    enabled: true
    schema_caching: true
  protobuf:
    enabled: true
```

### 4.3 消息压缩配置

| 压缩类型 | 压缩率 | CPU开销 | 适用场景 |
|----------|--------|---------|----------|
| none | 1x | 无 | 低延迟场景 |
| gzip | 5-10x | 中等 | 大消息、日志 |
| snappy | 2-4x | 低 | 高吞吐场景 |
| lz4 | 2-5x | 低 | 实时处理 |

配置策略：
- CDC同步消息：使用snappy压缩
- 日志类消息：使用gzip压缩
- 业务事件消息：不压缩（已有压缩字段）

## 6 消息可靠性设计

```yaml
# Kafka生产者配置

    # 重试配置

    max_in_flight_requests_per_connection: 5
    
    # 幂等性
    enable_idempotence: true
    
    # 压缩
    compression_type: snappy
    
    # 批处理
    batch_size: 16384
    linger_ms: 10
    buffer_memory: 33554432
    
    # 超时配置
    request_timeout_ms: 30000
    max_block_ms: 60000
```

### 6.2 消费者可靠性配置

```yaml

### 6.3 消息处理可靠性流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        消息处理可靠性流程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 消息接收                                                           │
│     ┌─────────┐    ┌─────────┐    ┌─────────┐                          │
│     │ Producer│───▶│  Kafka  │───▶│Consumer│                          │
│     └─────────┘    └─────────┘    └────┬────┘                          │
│                                         │                               │
│                                         ▼                               │
│  2. 消息处理                                                       │
│     ┌─────────────────────────────────────────────────────────┐        │
│     │  1. 接收消息并存储offset到内存                            │        │
│     │  2. 执行业务逻辑                                          │        │
│     │  3. 业务成功 ──▶ 提交offset                               │        │
│     │  4. 业务失败 ──▶ 记录失败原因 ──▶ 跳过/重试 ──▶ 提交offset │        │
│     └─────────────────────────────────────────────────────────┘        │
│                                         │                               │
│                                         ▼                               │
│  3. 失败处理                                                       │
│     ┌─────────────────────────────────────────────────────────┐        │
│     │  重试策略:                                               │        │
│     │  - 第1次失败: 立即重试                                   │        │
│     │  - 第2次失败: 等待1秒后重试                              │        │
│     │  - 第3次失败: 等待3秒后重试                              │        │
│     │  - 超过最大重试: 发送到死信队列(DLQ)                      │        │
│     └─────────────────────────────────────────────────────────┘        │
│                                         │                               │

## 7 消息顺序性设计

### 7.1 顺序性保障策略

| 场景 | 保障级别 | 实现方式 |
|------|----------|----------|
| 同分区有序 | 强顺序 | 单分区单消费者 |
| 同资产有序 | 业务顺序 | 按资产ID分区 |
| 全局有序 | 弱顺序 | 不保证（性能代价大） |
| 因果顺序 | 最终因果 | 消息携带因果ID |

### 7.2 分区策略

```java
// 按资产ID哈希分区（保证同一资产的变更有序）
@Component
public class AssetMessagePartitioner implements Partitioner {
    
    @Override
    public int partition(String topic, Object key, byte[] keyBytes, 
                         Object value, byte[] valueBytes, Cluster cluster) {
        String keyString = (String) key;
        
        if (keyString.startsWith("asset:")) {
            // 提取资产ID进行哈希
            String assetId = keyString.substring(6);
            int partitions = cluster.partitionsForTopic(topic).size();
            return Math.abs(assetId.hashCode() % partitions);
        }
        
        // 默认按消息键哈希
        return Math.abs(key.hashCode() % cluster.partitionsForTopic(topic).size());
    }
}

// 分区键设计
public class MessageKeyBuilder {
    
    // 资产变更事件 - 按资产ID分区
    public static String assetKey(String assetId) {
        return "asset:" + assetId;
    }
    
    // 血缘变更事件 - 按源资产ID分区
    public static String lineageKey(String sourceAssetId) {
        return "lineage:" + sourceAssetId;
    }
    
    // 质量检测事件 - 按资产ID分区
    public static String qualityKey(String assetId) {
        return "quality:" + assetId;
    }
    
    // 审计事件 - 按用户ID分区
    public static String auditKey(String userId) {
        return "audit:" + userId;
    }
}
```

### 8.2 消息积压处理策略

| 场景 | 症状 | 处理策略 |
|------|------|----------|
| 消费者能力不足 | lag持续增长 | 增加消费者实例、并行度 |

| 消费超时 | 频繁rebalance | 优化处理逻辑、增加超时时间 |
| 代码Bug | 消费失败 | 紧急修复、回滚、跳过消息 |

### 9.3 常见问题处理

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Producer发送超时 | Broker负载高 | 增加Broker、提升硬件、调整超时时间 |
| 消费者Rebalance频繁 | Session超时短 | 增加session.timeout.ms |
| 消息丢失 | acks配置不当 | 设置acks=all |
| 消息重复 | 幂等性未开启 | 开启enable.idempotence |
| 磁盘使用率高 | 保留策略过宽 | 收紧保留时间、开启压缩 |
| Offset消费位置错误 | 手动提交异常 | 审查提交逻辑、启用自动提交 |

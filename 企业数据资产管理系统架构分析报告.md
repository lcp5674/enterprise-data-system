### 2.3 信息架构补充建议

#### 【补充8】模块边界定义

**当前状态**：部分模块边界模糊，存在功能交叉。

**建议明确**：

| 模糊边界 | 归属界定 | 协调机制 |
|---------|---------|---------|
| 元数据管理 vs 数据地图 | 元数据管理负责元数据的增删改查API<br>数据地图负责前端展示和交互 | 数据地图调用元数据服务API |
| 数据血缘 vs 任务运维 | 数据血缘负责血缘关系存储和查询<br>任务运维负责任务资产管理和调度 | 血缘服务消费任务事件 |
| 数据质量 vs 数据标准 | 数据质量负责检测和评分<br>数据标准负责规则定义 | 质量规则引用标准 |
| 资产评分 vs 健康度 | 资产评分负责多维评分模型<br>健康度负责生命周期预警 | 健康度依赖评分结果 |

#### 【补充9】模块依赖治理规则

**建议补充依赖治理原则**：

1. **禁止循环依赖**
   ```
   A → B → C → A  (禁止)
   A → B → C → D  (允许)
   ```

2. **依赖层级限制**
   - 上层可调用下层服务
   - 下层禁止调用上层服务
   - 同层服务通过事件通信

3. **跨域调用审批**
   - 跨域直接调用需架构委员会审批
   - 优先通过事件总线解耦
